# SPRING BOOT — INTERVIEW REVISION GUIDE

## PAGE 3: DEPENDENCY INJECTION & BEAN MANAGEMENT

---

# QUICK RECAP

- Spring Boot is built on Spring Framework's IoC container.
- IoC container manages beans and their lifecycle.
- Dependency Injection (DI) is a core principle for achieving loose coupling and testability.

---

# 1. IOC CONTAINER (INVERSION OF CONTROL)

IoC means the control of creating and managing objects is transferred from the application to the Spring container.

---

## How It Works

1. You define beans using:
   - Annotations
   - XML
   - Java configuration

2. Spring container reads configuration and classpath.

3. It creates and manages beans.

4. It manages:
   - Lifecycle
   - Dependencies
   - Wiring

5. Application requests beans from container.

6. Container provides ready-to-use beans.

---

## Benefits of IoC

- Loose coupling
- Easier testing
- Better maintainability
- Centralized bean management

---

# 2. BEAN LIFECYCLE

## Lifecycle Steps

1. Bean Definition
   - From `@Component`, `@Bean`, etc.

2. Instantiation
   - Object is created

3. Populate Properties
   - Dependency Injection via setter/field/constructor

4. Aware Interfaces
   - `BeanNameAware`
   - `ApplicationContextAware`

5. Bean Post Processors
   - Before initialization

6. Initialization
   - `@PostConstruct`
   - `InitializingBean`

7. Bean Ready to Use

8. Destruction
   - `@PreDestroy`
   - `DisposableBean`

> Note: Beans are singleton by default.

---

# 3. STEREOTYPE ANNOTATIONS (BEAN DECLARATION)

These annotations make classes Spring-managed beans.

| Annotation | Purpose | Used In |
|---|---|---|
| `@Component` | Generic stereotype for Spring-managed component | Any Layer |
| `@Service` | Service layer bean (business logic) | Service Layer |
| `@Repository` | DAO layer bean + exception translation | Data Access Layer |
| `@Controller` | MVC Controller | Web / MVC Layer |
| `@RestController` | Combination of `@Controller` + `@ResponseBody` | REST API Layer |

---

## Key Point

All these annotations are specialized forms of `@Component`.

When Spring scans components, it registers them as beans.

---

# 4. AUTOWIRING & INJECTION TYPES

Spring can inject dependencies in multiple ways.

| Injection Type | Example | Description | Recommended |
|---|---|---|---|
| Constructor Injection | `@Autowired MyService(Repo r)` | Immutable & testable | ✅ |
| Setter Injection | `setRepo(Repo r)` | Optional dependencies | ⚠️ |
| Field Injection | `@Autowired private Repo repo;` | Direct field injection | ❌ |

---

## Best Practice

- Prefer Constructor Injection
- Avoid Field Injection (harder to test)

---

# 5. @AUTOWIRED

`@Autowired` tells Spring to inject a bean automatically.

---

## Example

```java
@Service
public class UserService {

    private final UserRepository repo;

    @Autowired
    public UserService(UserRepository repo) {
        this.repo = repo;
    }
}
```

---

## Notes

- Not required on constructor with single dependency (Spring Boot 2.6+)
- Can be used on:
  - Constructors
  - Setters
  - Fields
  - Methods

- By default, injection happens by type.

---

# 6. @QUALIFIER & @PRIMARY

Used when multiple beans of same type exist.

---

## @Qualifier Example

```java
@Service
public class PaymentService {

    private final PaymentGateway gateway;

    public PaymentService(
            @Qualifier("razorpay")
            PaymentGateway gateway) {

        this.gateway = gateway;
    }
}
```

---

## @Primary Example

```java
@Primary
@Component
public class RazorpayGateway implements PaymentGateway {
}
```

---

## Rule

If both `@Qualifier` and `@Primary` are present, `@Qualifier` wins.

---

# 7. BEAN SCOPES

| Scope | Annotation | Description | Use Case |
|---|---|---|---|
| Singleton (Default) | `@Scope("singleton")` | One bean instance per container | Stateless services |
| Prototype | `@Scope("prototype")` | New bean every request | Stateful beans |
| Request | `@Scope("request")` | One bean per HTTP request | Web layer |
| Session | `@Scope("session")` | One bean per HTTP session | Session data |
| Application | `@Scope("application")` | One bean per ServletContext | App-level data |
| WebSocket | `@Scope("websocket")` | One bean per WebSocket session | WebSocket apps |

---

## Note

Custom scopes can also be created if needed.

# 8. BEAN INITIALIZATION & DESTROY METHODS

---

## Initialization

### Using `@PostConstruct`

```java
@PostConstruct
public void init() {
    // initialization code
}
```

### Using `InitializingBean`

```java
@Override
public void afterPropertiesSet() {
    // init code
}
```

---

## Destruction

### Using `@PreDestroy`

```java
@PreDestroy
public void cleanup() {
    // cleanup code
}
```

### Using `DisposableBean`

```java
@Override
public void destroy() {
    // cleanup code
}
```

---

## Common Usage

Used to:
- Open/close resources
- Database connections
- File handling
- Cache cleanup

---

# 9. CIRCULAR DEPENDENCY

Occurs when:

- Bean A depends on Bean B
- Bean B depends on Bean A

---

## Example

```text
Bean A → Bean B
Bean B → Bean A
```

---

## How to Resolve

- Redesign application structure
- Use `@Lazy`
- Use Setter Injection
- Reduce tight coupling

---

## Warning

Avoid circular dependency because it:
- Increases complexity
- Makes maintenance difficult

# CHEAT SHEET (DI & BEANS)

| Task | Annotation |
|---|---|
| Create Bean | `@Component`, `@Service`, `@Repository`, `@Controller`, `@RestController`, `@Bean` |
| Inject Bean | `@Autowired` |
| Select Specific Bean | `@Qualifier("beanName")` or `@Primary` |
| Define Scope | `@Scope("singleton")` |
| Init Method | `@PostConstruct`, `InitializingBean` |
| Destroy Method | `@PreDestroy`, `DisposableBean` |

---

# KEY TAKEAWAYS

- Spring IoC container manages beans and lifecycle.
- Dependency Injection reduces coupling and improves testability.
- Prefer Constructor Injection over Field Injection.
- Proper bean scopes improve design quality.
- Good bean design leads to scalable and maintainable applications.

---

# BEST PRACTICES

- Use Constructor Injection for mandatory dependencies.
- Keep beans stateless whenever possible.
- Prefer composition over inheritance.
- Avoid circular dependencies.
- Follow Single Responsibility Principle (SRP).
